# Codex 交接稿：Toyota 車輛獎金計算器

## 一句話目標

建立一個給台灣 Toyota 一線業務使用的「單筆成交獎金計算器」：選車款與等級後，輸入階段、累計、所支援金、保險乙式、影音等資訊，立刻看總獎金。

## 使用者背景

使用者打算把工具放到 GitHub，提供上百位業務同仁使用。第一版希望快速可用、容易維護，暫時不需要後端、不需要帳號登入。

## 已確認產品決策

| 問題 | 決策 |
|---|---|
| 主要計算類型 | 單筆成交：選一台車立刻看總獎金 |
| 車款資料 | 預載目前台灣 Toyota 車系與等級，使用者再調金額 |
| 獎金設定 | 車輛獎金固定在資料檔；其他獎金可調 |
| 部署方式 | GitHub Pages 靜態網站 |
| 使用者規模 | 上百位業務同仁 |
| 主要裝置 | 手機優先，桌機也要可用 |

## 獎金組成

1. 車輛獎金
   - 來源：車款等級資料檔。
   - 模式：固定金額或百分比。
2. 階段獎金
   - 來源：規則檔級距。
   - 允許單筆覆寫。
3. 業代累計獎金
   - 來源：規則檔級距。
   - 允許單筆覆寫。
4. 所支援金
   - 現場手動輸入。
5. 保險乙式
   - 勾選後加計規則檔金額。
6. 影音
   - 勾選後加計規則檔金額。
7. 其他調整
   - 可正可負，用於臨時調整。

## 目前 UI 邏輯

桌機版：

- 左側：成交資料輸入。
- 右側：獎金試算結果與明細。
- 上方：清空重算、複製結果。

手機版：

- 先顯示成交資料輸入。
- 往下滑顯示獎金結果。
- 待討論：是否將總獎金移到上方或做成浮動結果列。

## 架構建議

### 第一版建議維持靜態網站

原因：

- GitHub Pages 可直接部署。
- 無後端維護成本。
- 業務同仁打開連結即可使用。
- 目前需求仍以單筆即時計算為主。

### 不建議第一版直接上複雜後台

原因：

- 會增加登入、權限、資料庫、資安與維護問題。
- 若真實獎金資料敏感，公開 GitHub Pages 不適合直接存放。
- 可以先用假資料或零元資料驗證介面與流程。

### 未來可演進方向

1. `bonus-engine.js`
   - 將所有計算邏輯抽離 DOM。
   - 方便寫測試與日後重構。
2. `schema/vehicles.schema.json`
   - 驗證車輛資料格式。
3. `schema/rules.schema.json`
   - 驗證級距、加項與支援金規則。
4. `tools/convert-bonus-sheet.js`
   - 將內部 Excel/CSV 轉成 JSON。
5. 私有部署或公司內網
   - 若要放真實獎金資料，應考慮需要權限控管的環境。

## 建議 Codex 第一輪要做的事

請先不要大改 UI 或改用框架。第一輪請專注於穩定性與可維護性：

1. 檢查資料檔路徑是否一致。
2. 檢查 README 與實際檔案名稱是否一致。
3. 抽出計算邏輯成純函式。
4. 新增最小測試。
5. 新增資料格式驗證。
6. 檢查手機版使用流程，提出是否需要浮動總獎金列。

## 驗收案例

### 案例 1：固定車輛獎金

- 車輛獎金：12,000
- 所支援金：3,000
- 其他項目：0
- 預期總獎金：15,000

### 案例 2：百分比車輛獎金

- 成交金額：1,000,000
- 車輛獎金百分比：1.2%
- 預期車輛獎金：12,000

### 案例 3：負調整

- 車輛獎金：12,000
- 所支援金：3,000
- 其他調整：-1,000
- 預期總獎金：14,000
