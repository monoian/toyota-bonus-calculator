# AGENTS.md — Codex 專案指令

請使用繁體中文（台灣）與使用者溝通，程式註解與文件也以繁體中文為主。

## 專案定位

這是「Toyota 台灣一線業務單筆成交獎金計算器」。目標使用者是上百位銷售同仁，主要在手機上操作，也會有桌機使用情境。第一版以 GitHub Pages 靜態網站部署，不需要後端、不需要登入、不需要資料庫。

## 已確認需求

- 主要計算情境：單筆成交，選一台車後立即看總獎金。
- 車款資料：預載台灣 Toyota 目前車系與等級，後續由使用者自行調整金額。
- 獎金設定：
  - 車輛獎金：固定在車輛資料 JSON 內，支援固定金額與百分比模式。
  - 階段獎金：可調，依級距或覆寫。
  - 業代累計獎金：可調，依級距或覆寫。
  - 所支援金：可手動輸入。
  - 保險乙式、影音：可勾選並依規則檔計算。
- 部署目標：GitHub Pages，供大量業務同仁使用。
- 重要限制：若 repository 或 GitHub Pages 是公開的，不能放公司敏感獎金資料。

## 現有技術架構

- 純前端：HTML + CSS + JavaScript。
- 無 build step、無框架、無後端。
- 車輛資料：`data/vehicles.tw.toyota.2026-06-21.json`
- 規則資料：`data/rules.example.json`
- 主畫面：`index.html`
- 樣式：`assets/styles.css`
- 邏輯：`assets/app.js`

## 開發原則

1. 優先維持簡單、可直接部署到 GitHub Pages。
2. 不要在程式碼或範例資料中加入真實公司敏感獎金。
3. 新增功能前，先確認是否會讓非技術人員難以維護。
4. 計算邏輯應盡量拆成純函式，方便測試。
5. JSON 結構要穩定，讓使用者能每月更新車輛獎金與級距規則。
6. 手機體驗優先，因為主要使用者是一線業務。
7. UI 文字要清楚、低學習成本，避免工程術語。

## 建議 Codex 優先任務

1. 檢查目前資料檔名稱、README 說明、app.js 路徑是否一致。
2. 將獎金計算邏輯從 DOM 操作中抽出成可測試的純函式。
3. 新增基本測試案例，涵蓋固定車輛獎金、百分比車輛獎金、階段獎金、累計獎金、加項與負調整。
4. 新增 JSON schema 或驗證腳本，避免車輛資料格式錯誤導致現場無法使用。
5. 評估手機版是否新增「總獎金浮動列」或將結果區塊提前。
6. 規劃未來的管理端或資料匯入流程，例如 CSV/Excel 轉 JSON，但第一版不要急著加入複雜後台。

## 驗收標準

- 在本機靜態伺服器或 GitHub Pages 上能正常載入。
- 使用者選車系與等級後，能看到建議售價與車輛獎金資料。
- 修改階段台數、累計台數、所支援金、保險乙式、影音後，總獎金即時更新。
- 手機寬度下版面可讀、可點、可輸入。
- `複製結果` 內容可貼到 LINE 或內部群組。
- 不引入真實內部獎金資料。
